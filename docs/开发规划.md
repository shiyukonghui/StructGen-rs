## 1. 依赖分析总览

通过对 `docs/` 目录下全部 8 份设计文档的审查，梳理出以下模块依赖关系图：

```
                          ┌─────────────────────────────────┐
                          │     需求规格说明书 (架构总纲)      │
                          └───────────────┬─────────────────┘
                                          │ 指导
                          ┌───────────────▼─────────────────┐
                          │       core (核心抽象层)           │
                          │   FrameState / SequenceFrame     │
                          │   Generator trait / CoreError    │
                          │   GenParams / OutputFormat       │
                          │   GeneratorRegistry              │
                          └───────┬───────┬───────┬─────────┘
                                  │       │       │
              ┌───────────────────┘       │       └───────────────────┐
              ▼                           ▼                           ▼
  ┌───────────────────────┐ ┌───────────────────────┐ ┌───────────────────────┐
  │  generators (生成器)   │ │  pipeline (后处理管道) │ │    sink (输出适配层)    │
  │  CA / L系统 / 物理     │ │  标准化/去重/差分/令牌  │ │  Parquet / Text / Binary │
  │  连续系统 / 算法轨迹   │ │  Processor trait       │ │  SinkAdapter trait       │
  └───────────┬───────────┘ └───────────┬───────────┘ └───────────┬───────────┘
              │                         │                         │
              └─────────────┬───────────┴─────────────┬───────────┘
                            ▼                         │
                  ┌───────────────────────┐            │
                  │  scheduler (任务调度)  │◄───────────┘
                  │  Manifest / Shard      │
                  │  run_manifest()        │
                  │  种子派生 / 并行执行    │
                  └───────────┬───────────┘
                              │
              ┌───────────────┼───────────────┐
              ▼               ▼               ▼
  ┌───────────────────────┐ ┌───────────────────────┐
  │ metadata (元数据监控)  │ │   CLI (命令行入口)     │
  │  JSON快照 / 进度追踪   │ │   参数解析 / 装配运行   │
  │  RunMetadata           │ │   main() 入口          │
  └───────────────────────┘ └───────────────────────┘
```

**关键依赖说明：**

| 模块 | 直接依赖 | 说明 |
|------|---------|------|
| **core** | 无（仅 std、serde、thiserror） | 系统最底层，定义所有共享类型与接口契约 |
| **generators** | core | 实现 `Generator` trait，使用 `FrameState`、`SequenceFrame`、`GenParams` |
| **pipeline** | core | 实现 `Processor` trait，消费/产出 `SequenceFrame` 迭代器 |
| **sink** | core | 实现 `SinkAdapter` trait，消费 `SequenceFrame` 并持久化 |
| **scheduler** | core + generators + pipeline + sink | 协调中枢：通过注册表调用生成器，组装管道链，驱动输出适配器 |
| **metadata** | core + scheduler（类型引用）+ sink（类型引用） | 汇总 `ShardResult` 中的 `OutputStats` 生成 `RunMetadata` |
| **CLI** | 全部模块 | 最终集成点：解析参数、注册组件、启动调度器、写入元数据 |

## 2. 分层开发计划

按照依赖自底向上的原则，将开发划分为 **五个阶段**。

### 第一阶段：基础设施 —— core 模块

**对应文档**：[core模块详细设计.md](./core模块详细设计.md)

**依赖**：无（仅 Rust 标准库 + serde + thiserror）

**开发内容**：

| 优先级 | 文件 | 内容 |
|--------|------|------|
| P0 | `src/core/frame.rs` | `FrameState` 枚举、`FrameData` 结构体、`SequenceFrame` 结构体 |
| P0 | `src/core/error.rs` | `CoreError` 枚举、`CoreResult<T>` 类型别名 |
| P0 | `src/core/params.rs` | `GenParams`、`GridSize`、`GlobalConfig`、`OutputFormat` 枚举 |
| P0 | `src/core/generator.rs` | `Generator` trait（含 `generate_stream`、`generate_batch` 默认实现） |
| P0 | `src/core/registry.rs` | `GeneratorRegistry` 结构体（名称→构造函数映射） |
| P1 | `src/core/mod.rs` | 统一重导出所有公开类型 |

**完成标准**：
- 所有类型定义完整，通过编译
- `FrameState` 的 `as_integer()`/`as_float()`/`as_bool()` 转换逻辑通过单元测试
- `GenParams` 的扩展字段序列化/反序列化往返测试通过
- `GeneratorRegistry` 的注册、查找、重复注册 panic 测试通过
- 无 `clippy` 警告

**预估工期**：基准工期，作为后续阶段参考

---

### 第二阶段：独立功能模块（可并行开发）

本阶段包含三个完全独立、相互无依赖的模块，**可以并行开发**。

#### 2.1 sink（输出适配层）

**对应文档**：[sink模块详细设计.md](./sink模块详细设计.md)

**依赖**：core（`SequenceFrame`、`FrameState`、`OutputFormat`、`CoreError`）

**开发内容**：

| 优先级 | 文件 | 内容 |
|--------|------|------|
| P0 | `src/sink/adapter.rs` | `SinkAdapter` trait、`OutputStats`、`OutputConfig`、`format_output_filename()` |
| P0 | `src/sink/factory.rs` | `SinkAdapterFactory` 类型别名 |
| P1 | `src/sink/parquet.rs` | `ParquetAdapter`：Arrow Schema 构建、RecordBatch 转换、Snappy 压缩写出 |
| P1 | `src/sink/text.rs` | `TextAdapter`：Unicode 字符映射、BufWriter 行写出 |
| P1 | `src/sink/binary.rs` | `BinaryAdapter`：固定头格式、帧序列化、mmap 友好布局 |
| P2 | `src/sink/mod.rs` | 模块根，重导出 |

**测试重点**：
- Parquet 写入→读取往返一致性
- TextAdapter 输出为合法 UTF-8
- BinaryAdapter 文件头 magic 校验
- 原子写入：成功后无 `.tmp` 残留文件
- 文件名格式唯一性

#### 2.2 pipeline（后处理管道层）

**对应文档**：[pipeline模块详细设计.md](./pipeline模块详细设计.md)

**依赖**：core（`SequenceFrame`、`FrameState`、`FrameData`、`CoreError`）

**开发内容**：

| 优先级 | 文件 | 内容 |
|--------|------|------|
| P0 | `src/pipeline/processor.rs` | `Processor` trait、`ProcessorFactory` 类型别名 |
| P0 | `src/pipeline/registry.rs` | `ProcessorRegistry`（名称→构造函数映射） |
| P1 | `src/pipeline/normalizer.rs` | `Normalizer`：Linear/LogBucket/UniformQuantile 三种方法 |
| P1 | `src/pipeline/dedup.rs` | `DedupFilter`：连续重复去重、全零帧过滤、低熵过滤 |
| P1 | `src/pipeline/diff_encoder.rs` | `DiffEncoder`：相邻帧差分，支持首帧零参考 |
| P1 | `src/pipeline/token_mapper.rs` | `TokenMapper`：整数值→Unicode 码点映射 |
| P2 | `src/pipeline/clip_stitcher.rs` | `ClipStitcher`：序列截断与分隔符插入 |
| P2 | `src/pipeline/null_proc.rs` | `NullProcessor`：透传处理器（测试用） |
| P2 | `src/pipeline/mod.rs` | 模块根，`register_all()` 注册全部内置处理器 |

**测试重点**：
- Normalizer Linear 缩放正确性
- DedupFilter 连续相同帧移除
- DiffEncoder 差分计算正确
- TokenMapper 输出码点在 Unicode 安全范围
- 多处理器链式组合端到端测试
- ProcessorRegistry 未知名称拒绝

#### 2.3 generators（生成器仓储）

**对应文档**：[生成器详细设计.md](./生成器详细设计.md)

**依赖**：core（`Generator` trait、`FrameState`、`SequenceFrame`、`GenParams`）

**开发内容**：

| 优先级 | 文件 | 内容 |
|--------|------|------|
| P0 | `src/generators/mod.rs` | 模块根，`register_all()` 注册所有生成器 |
| P1 | `src/generators/ca.rs` | `CellularAutomaton`：1D/2D CA，Wolfram 规则，周期/固定/反射边界 |
| P1 | `src/generators/lorenz.rs` | `LorenzSystem`：RK4 积分，参数化 σ/ρ/β |
| P1 | `src/generators/logistic.rs` | `LogisticMap`：离散混沌映射，参数 r 扫描 |
| P2 | `src/generators/nbody.rs` | `NBodySim`：多体引力，软化因子，显式欧拉/RK4 积分器 |
| P2 | `src/generators/lsystem.rs` | `LSystem`：上下文无关重写，龟图解释器可选 |
| P2 | `src/generators/vm.rs` | `AlgorithmVM`：精简栈式虚拟机，随机程序生成 |
| P3 | `src/generators/boolean_network.rs` | `BooleanNetwork`：随机布尔网络演化 |
| P3 | `src/generators/ifs.rs` | `IteratedFunctionSystem`：分形吸引子点序列 |
| P3 | `src/generators/formal_grammar.rs` | `FormalGrammar`：CFG 随机句子生成 |

**测试重点**：
- 确定性：固定种子→固定输出（前 N 帧二进制哈希比对）
- CA 规则 30/110 经典模式验证
- 洛伦兹系统吸引子形态基本正确
- 分片种子派生后产出互不重叠
- 参数校验：非法规则号/积分步长等应报错

### 第三阶段：调度协调 —— scheduler 模块

**对应文档**：[scheduler模块详细设计.md](./scheduler模块详细设计.md)

**依赖**：core + generators + pipeline + sink

**开发内容**：

| 优先级 | 文件 | 内容 |
|--------|------|------|
| P0 | `src/scheduler/seed.rs` | `derive_seed()` 确定性种子派生函数 |
| P0 | `src/scheduler/manifest.rs` | `Manifest`、`TaskSpec`、`GlobalConfig` 定义与 YAML 反序列化 |
| P0 | `src/scheduler/shard.rs` | `Shard`、`ShardResult` 定义，`shard_tasks()` 分片切分算法 |
| P0 | `src/scheduler/executor.rs` | `execute_shard()`：实例化生成器→组装管道→驱动适配器→收集统计 |
| P1 | `src/scheduler/mod.rs` | `run_manifest()` 主入口，rayon 并行调度，容错汇总 |

**测试重点**：
- `derive_seed` 确定性与 wrapping 行为
- 分片数量与样本分布正确（均匀/不均匀）
- 清单校验拒绝未知生成器、零样本数
- 单分片失败不影响其他分片（容错隔离）
- 相同清单两次运行→完全一致的 `ShardResult`
- Mock 生成器集成：验证端到端数据流

### 第五阶段：最终集成 —— CLI 模块

**对应文档**：[CLI模块详细设计.md](./CLI模块详细设计.md)

**依赖**：全部模块（core + generators + pipeline + sink + scheduler + metadata）

**开发内容**：

| 优先级 | 文件 | 内容 |
|--------|------|------|
| P0 | `src/main.rs` | `CliArgs` 定义（clap derive）、`main()` 入口、`run()` 装配函数 |
| P1 | `Cargo.toml` | 二进制目标配置 `[[bin]]`、所有依赖声明 |

**`run()` 函数核心流程**：

```
1. 解析命令行参数 (clap)
2. 初始化日志 (metadata::init_logger)
3. 加载并校验 YAML 清单 (serde_yaml + scheduler::Manifest)
4. 命令行参数覆盖清单全局配置
5. 设置 rayon 线程池大小
6. 初始化生成器注册表 (generators::register_all)
7. 初始化处理器注册表 (pipeline::register_all)
8. 校验清单中引用的生成器/处理器均已注册
9. 创建输出目录
10. 创建进度追踪器 (metadata::ProgressTracker)
11. 启动进度显示线程（可选）
12. 记录开始时间，调用 scheduler::run_manifest()
13. 等待完成，停止进度显示
14. 写入元数据 (metadata::write_metadata)
15. 打印汇总报告
16. 返回退出码 (0=成功, 1=部分失败, 2=完全失败)
```

**测试重点**：
- `--help` 输出包含所有参数
- 缺少 `--manifest` 时报错退出
- YAML 解析失败时输出可读错误信息
- 端到端测试：最小 YAML 清单 → 产出文件 + metadata.json
- 退出码：全成功=0，部分失败=1，全失败=2

## 5. 风险与注意事项

### 5.1 技术风险

| 风险 | 影响模块 | 缓解措施 |
|------|---------|----------|
| Arrow/Parquet crate API 变动 | sink | 锁定主版本号，阶段二优先验证 |
| rayon 线程池与 tracing 异步日志冲突 | scheduler + metadata | 阶段三集成测试覆盖并发场景 |
| 浮点确定性跨平台差异 | generators | CI 覆盖 x86_64 和 aarch64，文档注明平台要求 |
| YAML 清单格式演进导致旧文件不兼容 | scheduler + CLI | Manifest 增加 version 字段，支持迁移 |

### 5.2 开发建议

1. **阶段一完成后立即编写集成测试骨架**：用一个 mock 生成器验证 core → scheduler 的类型通路。
2. **阶段二的三个模块建议分配不同开发者并行推进**，互不阻塞。每周同步接口约定。
3. **每个阶段完成后打 tag**，便于回溯。例如 `v0.1.0-core`、`v0.2.0-modules`、`v0.3.0-scheduler`。
4. **generators 模块按优先级分批交付**：P1（CA + 洛伦兹 + 逻辑斯蒂）先交付以打通全链路，P2/P3 后续迭代补充。
5. **测试覆盖率目标 ≥ 80%**：core 和 scheduler 模块应达到 90%+，generators 中确定性测试应覆盖所有生成器类型。
