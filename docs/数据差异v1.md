分析结果：NCA 数据格式兼容性比较
一、Rust 项目 NCA 生成器 vs 参考项目 ca-pre-pretraining-main
1. 核心算法差异
维度	Rust nca2d.rs	Python 参考 utils/nca.py
NN 架构	自定义：conv3x3 → fc1 → ReLU → fc2	Flax：Conv(3x3,4) → Conv(1x1,16) → ReLU → Conv(1x1,d_state*n_groups)
参数量	~584 参数 (默认 d_state=10)	Flax 默认初始化，参数量不同
权重初始化	PRNG 均匀 [-1, 1]	Flax 默认 (Lecun Normal / Zero bias)
Conv padding	手动 wrap padding (环面)	jnp.pad(pad_width=1, mode='wrap') + VALID
中间层	conv_features → hidden_dim → channels	4 → 16 → d_state*n_groups
采样方式	手动 CDF softmax 采样	jax.random.categorical
temperature 默认	0.0 (argmax)	1.0 (随机采样)
关键问题：网络架构不同。Rust 实现是分拆的 conv3x3 → fc1 → fc2，而 Python 是 Conv(3x3) → Conv(1x1) → ReLU → Conv(1x1)。即使参数对齐，由于 Flax Conv 与手动 conv3x3 的等效性取决于权重排列方式，两者不会产生相同输出。
2. 数据格式差异
Rust 输出格式 (Parquet)：
Schema: sample_id: Int64?, step_index: Int64, state_dim: Int32, state_values: Binary, label: Utf8?
state_values: 每个 FrameState 为 9 字节 (type_tag + 8字节payload)
NCA 值为 FrameState::Integer(u8 as i64)，即 tag=0x01, 范围 [0, d_state-1]
Python 参考 (内存张量)：
Shape: (num_sims, num_examples, H, W, C) — 直接 JAX ndarray
值为 jnp.int32，范围 [0, d_state-1]
通过 NCA_Tokenizer.encode_task() 编码为 (B, N, grid_len) 的 token 序列
Token 包含：start_token + patch_tokens + end_token，patch 大小默认可配
3. 兼容性评估
无法无缝兼容，原因如下：
架构不对齐：Rust 的 conv3x3 → fc1 → fc2 不等价于 Python 的 Conv(3x3,4) → Conv(1x1,16) → ReLU → Conv(1x1)。结构上 Rust 分为独立的卷积和全连接层，而 Python 的第二个 Conv(1x1,16) 本质是逐点卷积（等价于 fc），但通道数和激活函数位置不同。
权重初始化不同：Rust 用 [-1, 1] 均匀分布，Python 用 Flax 默认的 LeCun Normal 初始化。即使同种子也不会产生相同权重。
temperature 默认值不同：Rust 默认 0.0（确定性 argmax），Python 默认 1.0（随机采样）。这导致即使权重相同，输出行为也不同。
数据管道不同：Python 端直接在内存中操作 JAX 张量，经 NCA_Tokenizer 做 patch tokenization 后喂入 Transformer；Rust 端序列化为 Parquet 二进制格式，需要额外编写读取/解码/tokenization 逻辑才能对接训练。
Tokenization 缺失：Rust 项目没有 NCA_Tokenizer 的等价实现。参考项目将 (H, W) 网格 patch 化为 token 序列（含 start/end token），这是训练的关键预处理步骤。
二、其他生成器的数据训练兼容性
生成器	FrameState 类型	状态维度	与 NCA 训练兼容性
CellularAutomaton (1D CA)	Bool	128 (width)	低 — 二值 Bool，而 NCA 是多值 Integer；且维度差异大
CA2D (2D CA)	Integer 或 Bool	rows×cols	中 — 同为 2D 网格，若 d_state 匹配可共享 tokenizer；但传统 CA 规则固定，缺乏规则多样性
CA3D	Integer 或 Bool	比较大	低 — 3D 结构与 2D NCA 训练不兼容
LorenzSystem	Float	3 (x,y,z)	极低 — 连续浮点 3D 状态，与离散 2D 网格完全不同
LogisticMap	Float	1	极低 — 连续标量序列
NCA2D	Integer	rows×cols×n_groups	高（设计目标） — 与参考项目格式语义一致，但需要适配层
三、关键差距与建议
要使 Rust NCA2D 数据能被参考项目训练流程使用，需要解决：
网络架构对齐：当前 Rust 实现的 3 层结构 (conv3x3→fc1→fc2) 与 Python 的 4 层结构 (Conv3x3→Conv1x1→ReLU→Conv1x1) 不等价。如果要复现参考项目行为，需要将 Rust 的 NcaWeights 重构为 Conv3x3 → Conv1x1(hidden) → ReLU → Conv1x1(output) 的结构。
权重初始化对齐：需要用与 Flax 相同的初始化分布（LeCun Normal），而非 [-1,1] 均匀。
Parquet→JAX 数据桥接：需要编写 Python 端的 Parquet 读取器，将 state_values 解码为 (H, W, C) 的 ndarray，并经 NCA_Tokenizer 编码。
Tokenization 层：Rust 端不包含 patch tokenization，这是训练侧的需求，可以在 Python 端补齐。
总结：两者不能无缝兼容。核心障碍是 NN 架构不同导致的数值不等价，以及数据输出格式（Parquet 二进制 vs 内存 JAX 张量）和训练管道（缺少 Tokenization 适配）的差异。Rust NCA2D 在语义层面与参考项目对齐（都是 2D 多状态 NCA），但需要做架构对齐和数据桥接才能实际用于训练。
