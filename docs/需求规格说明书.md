以下是 **StructGen-rs** 系统的架构规格说明书，它描述了如何用纯 Rust 构建一个高性能、可扩展的程序化数据生成器，专门用于生产“非语言、富结构”的训练语料。

## 1. 引言

### 1.1 目的
本文档旨在详细规定 **StructGen-rs** 的架构设计、模块划分及功能逻辑。该系统用于程序化生成多种类型的非自然语言、具有内在规则与时空结构的数据，其核心目标是为大型语言模型的预训练提供高质量、多样化的合成语料。

### 1.2 设计目标
- **高性能**：充分利用 Rust 的编译优化与现代 CPU 的多核并行能力，保证大批量数据生成的高吞吐量。
- **可扩展性**：生成器以插件形式组织，支持随时添加新的数据类型而不影响现有系统。
- **可复现性**：通过确定性的种子管理与完整的元数据记录，确保任何数据集均可精确重构。
- **低资源开销**：采用流式处理与批处理结合的架构，支持 TB 级数据集生成而不发生内存溢出。
- **易用性**：提供简洁的命令行接口，通过单一配置文件驱动整个生成流程。

## 2. 总体架构

StructGen-rs 采用分层架构，由一个二进制入口和多个功能库构成，通过核心抽象接口衔接。系统整体分为六个主要功能层：**核心抽象层、生成器层、任务调度层、后处理管道层、输出适配层、元数据与监控层**。

```
┌──────────────────────────────────────┐
│          CLI 入口 (二进制)            │  解析参数、装配并启动流水线
├──────────────────────────────────────┤
│  生成器注册中心 & 统一接口 (core)     │  定义 Generator trait 与公共数据类型
├──────────────────────────────────────┤
│  ┌──────────────────────────────┐    │
│  │    任务调度与分片 (scheduler)│    │  参数解析、种子派生、并行任务分发
│  └──────────────────────────────┘    │
│  ┌──────────────────────────────┐    │
│  │     生成器仓储 (generators)  │    │  CA、物理、算法、L系统等具体实现
│  └──────────────────────────────┘    │
│  ┌──────────────────────────────┐    │
│  │    后处理管道 (pipeline)     │    │  标准化、去重、分词映射等
│  └──────────────────────────────┘    │
│  ┌──────────────────────────────┐    │
│  │     输出适配器 (sink)        │    │  Parquet/Arrow、纯文本等格式写出
│  └──────────────────────────────┘    │
├──────────────────────────────────────┤
│      元数据与日志服务 (metadata)      │  参数快照、哈希记录、运行日志
└──────────────────────────────────────┘
```

在运行时，数据以流的形式依次通过生成器、后处理管道和输出适配器各层。所有计算密集型任务均在同一个 Rust 进程中完成，无需跨语言交互。

## 3. 模块详细设计逻辑

### 3.1 核心抽象层 (core)

本层定义了系统中所有可共享的数据结构与核心接口，是其余各层依赖的基础。

#### 3.1.1 数据类型
- **FrameState**：单个状态值的标记联合体，统一承载整型（`i64`）、浮点型（`f64`）和布尔型（`bool`）三种互斥语义。`Float` 变体持有有限性不变量（非 NaN、非 ±Infinity），生成器应通过受检构造函数 `FrameState::float()` 创建浮点值。`as_float()` 对 `Integer` 变体执行精确范围检查，仅当 `|v| ≤ 2^53` 时转换，超出范围返回 `None`，防止静默精度丢失。
- **FrameData**：表示单个时间步的状态数据，内部为 `Vec<FrameState>`。实现标准 `FromIterator` trait，支持 `.collect::<FrameData>()` 惯用写法。
- **SequenceFrame**：一个时间步的完整帧，包含时间步索引、状态数据以及可选的文本标签（用于描述该帧规则或语义，便于后续混合训练）。
- **GenParams**：通用的参数载体，包含所有生成器共用字段（如序列长度、网格尺寸），并提供一个动态字段容器用于携带生成器特定的扩展参数。
- **LogLevel**：日志级别枚举（Trace/Debug/Info/Warn/Error），替代自由字符串，在类型层面排除无效值。JSON 序列化时使用小写形式（`"trace"`、`"debug"` 等）。

#### 3.1.2 生成器接口 (Generator trait)
所有生成器必须实现一个统一的接口，该接口至少包含以下能力：
- **名称识别**：返回一个唯一标识该生成器的字符串名称。
- **流式生成**：返回一个 `'static` 生命周期的惰性迭代器（`+ Send`），在迭代过程中逐步产生帧数据。该方式是默认推荐的方式，可有效控制内存峰值。迭代器不能借用 `&self`，实现者需将必要状态克隆或移入闭包。
- **批量生成**：接收一个随机种子与通用参数，输出完整的帧序列。该方式要求 `seq_length > 0`，否则返回参数错误（因为无限流无法收集为 Vec），适用于数据量可控的场景。

生成器的构造不通过 trait 方法完成，而是通过注册在 `GeneratorRegistry` 中的工厂函数进行。

#### 3.1.3 生成器注册与配置校验
- 提供一个全局的"生成器注册表"，该表为一个从名称字符串到生成器构造函数的映射。注册时若名称重复，返回错误而非 panic，使调用方可在运行时决定处理策略。
- `GlobalConfig` 和 `GridSize` 提供 `validate()` 校验方法，在配置入口处检查 `num_threads`、`output_dir`、`shard_max_sequences`、`rows`、`cols` 等字段的合法性，将配置错误尽早暴露。

### 3.2 任务调度层 (scheduler)

调度层负责将用户需求转化为可执行的生成任务，并管理并行执行与资源分配。

#### 3.2.1 清单解析
- 从 YAML 格式的清单文件中读取任务列表，每个任务指定生成器名称、所需参数、目标样本数量以及可选的流水线配置。
- 解析后的任务被转换为内部的任务描述结构，包含所有必要信息。

#### 3.2.2 种子管理与分片
- 为保证可复现性，用户可为每个任务提供一个基础种子。
- 调度器将单个任务的样本数量 `N` 按设定的分片大小（或根据 CPU 核心数自动计算）切分为多个分片。每个分片分配 `M` 个样本。
- 每个分片获得一个唯一的派生种子，派生规则为：`种子 = 基础种子 + 分片序号`（或通过加密哈希派生），保证不同任务、不同分片之间的种子无冲突且可重现。

#### 3.2.3 并行执行逻辑
- 调度器将所有分片任务收集为一个任务队列。
- 利用数据并行库（如 rayon）在线程池上并发执行每个分片。每个分片的执行逻辑为：实例化对应生成器，调用其流式生成方法，将产生的帧序列传递给后处理管道和输出适配器。
- 执行过程支持两种模式：
    - **阻塞收集模式**：所有分片结果收集到内存后统一写出，适合中小规模数据。
    - **流式写出模式**：每个分片生成的同时即异步或同步写入磁盘，通用通道或专用写入线程避免争用，确保内存占用恒定。

### 3.3 生成器仓储 (generators)

该层是结构化数据的来源，包含所有具体生成器的实现。每个生成器作为一个独立的模块存在，但都通过注册机制暴露给调度器。

#### 3.3.1 元胞自动机生成器
- 逻辑：根据给定的规则号（Wolfram 规则或生命游戏规则等）、网格尺寸和演化步数，迭代更新网格状态。
- 状态表示：使用位平面或多维布尔数组，演化逻辑通过位运算或邻居查表实现。
- 输出：按时间步顺序输出网格的完整状态或展开的一维向量。

#### 3.3.2 连续动力系统生成器
- 逻辑：实现经典混沌映射（如逻辑斯蒂映射、洛伦兹方程、Henon 映射）的数值迭代。
- 支持参数扫描：可在指定参数范围内（如 r ∈ [3.5, 4.0]）均匀采样，生成不同动力学行为（周期、混沌）的序列。
- 输出：一维或多维浮点状态向量序列。

#### 3.3.3 物理模拟生成器
- 逻辑：基于简化的物理规则（如多体引力、双摆）进行数值积分，产生粒子位置、角度等状态随时间的变化。
- 积分方法：采用显式欧拉法或RK4，并支持批量并行计算多个独立系统，共用相同的物理参数但不同的初始条件。
- 输出：每一帧为所有粒子的位置与速度向量。

#### 3.3.4 算法轨迹生成器
- 逻辑：内嵌一个简化的栈式虚拟机或图灵机。定义一套指令集，随机生成程序，然后逐步执行并记录内存、寄存器或纸带的状态快照。
- 可选规则：可生成排序、搜索等简单算法的逐步执行轨迹，暴露中间变量变化。
- 输出：每个执行步骤的机器状态快照。

#### 3.3.5 L-系统生成器
- 逻辑：实现确定的、上下文无关的L-系统重写规则。定义公理、产生式规则和迭代次数，将符号串逐次展开。
- 状态表示：符号用单字节编码，规则用符号到扩展序列的映射。支持参数化符号（龟图解释器）的状态序列输出，即生成绘制图形的动作序列。
- 输出：每个迭代步生成的符号序列，或龟图解释器生成的坐标/角度序列。

#### 3.3.6 其他生成器
- 包括但不限于形式语法生成器、棋谱序列生成器、逻辑约束系统（如斑马谜题生成器）、分形迭代函数系统等。所有生成器遵循相同接口规范。

#### 3.3.7 生成器注册机制
- 提供一个静态的“生成器注册表”，该表为一个从名称字符串到生成器构造函数的映射。
- 添加新生成器时，只需在对应模块中实现接口，并在注册表中添加一条映射条目，无需改动其他框架代码。

### 3.4 后处理管道 (pipeline)

管道层负责对原始生成帧进行清洗、变换和适配，使其更适用于语言模型训练。

#### 3.4.1 处理器接口
- 每个后处理步骤实现一个统一的处理器接口，接受帧序列，返回处理后的帧序列。
- 处理器可串行级联，用户可在清单中指定按顺序应用的一系列处理器。

#### 3.4.2 内置处理器
- **标准化器**：将连续浮点值通过线性缩放、对数分桶或均匀量化等方式映射到有限整数范围（如0–255），形成固定大小的词汇表。
- **去重与过滤**：移除连续重复帧（静止状态）或全零帧，可选地按熵值过滤掉低复杂度序列，保证数据多样性。
- **差分编码器**：对相邻帧计算差分并输出，可大幅降低缓慢变化序列的冗余，同时保留变化模式。
- **令牌映射器**：将离散状态值直接映射为 Unicode 或单字节令牌，使其能够直接作为语言模型的输入文本。例如，将每个0–255的值映射到一个专用字符。
- **序列截断与拼接**：将过长序列切分为固定长度片段，或为不同序列添加分隔符。

### 3.5 输出适配层 (sink)

适配层负责将处理后的帧序列转换为特定格式，并写入持久存储。

#### 3.5.1 适配器接口
- 适配器拥有初始化、逐批写入帧以及完成（关闭文件、写入页脚）的方法。
- 支持多输出流：同一任务可同时输出到多种格式（需要时可通过复制流实现）。

#### 3.5.2 支持的输出格式
- **Apache Parquet**：列式存储格式，适合大规模数值序列的分析与快速加载。内部按时间步、状态维度等分列，利用压缩与编码降低存储量并加速随机读取。
- **内存映射二进制文件**：原始二进制转储，直接映射到内存地址空间，适合后续高效随机访问训练。
- **纯文本文件**：将令牌映射后的 Unicode 序列写出为文本文件，可选择分片，每个分片包含固定行数或字节数，可直接被语言模型数据加载器读取。

#### 3.5.3 分片策略
- 根据输出文件大小或样本数量自动分片，如每个分片文件包含最多 10,000 条序列。
- 文件名嵌入分片索引与基础种子，保证唯一性。

### 3.6 元数据与监控层 (metadata)

本层确保生成过程的可追溯性与可复现性。

#### 3.6.1 元数据记录
- 每个生成任务完成后，生成一个独立的元数据文件（如 JSON 格式），包含：
    - 任务配置快照（生成器名称、参数、种子基础值、样本数量）。
    - 数据统计（总帧数、总字节数、每样本平均长度）。
    - 数据完整性校验信息（如整个输出文件的 SHA-256 哈希值）。
    - 软件版本信息（StructGen-rs 版本、依赖库版本摘要）。
- 元数据文件与数据文件放在同一目录下，命名关联。

#### 3.6.2 日志与进度上报
- 系统内置结构化日志，记录生成进度（已生成样本数/总样本数）、耗时、错误信息等。
- 日志输出可配置为控制台、文件或同时，并支持不同级别的详细程度。

### 3.7 命令行接口 (CLI)

CLI 是用户与系统交互的唯一入口，负责参数解析与工作流装配。

- 输入参数：
    - `--manifest`：任务清单 YAML 文件的路径。
    - `--output-dir`：输出根目录。
    - `--format`：输出格式，可选 `parquet`, `text`, `binary`。
    - `--num-threads`：指定并行线程数（默认为 CPU 逻辑核心数）。
    - `--log-level`：日志级别。
- 执行流程：
    1. 解析命令行参数，验证清单和输出目录的有效性。
    2. 加载清单，构造调度器并注入所有已注册的生成器。
    3. 启动生成流水线，实时输出进度信息。
    4. 等待所有任务完成，输出汇总报告，并退出。

## 4. 数据流与并发模型

系统遵循“任务分片→并行生成→后处理→写出”的流水线模型。数据流为单向、多路并行：

1. **分片分配**：调度器将任务拆分为独立分片，每个分片负责生成一定数量的样本。
2. **生成与处理**：每个分片被调度到线程池中的一个线程。线程内依次执行：通过生成器产生一个样本序列；将序列送入后处理管道链；将处理后的帧批次传递给输出适配器。
3. **写出同步**：当使用流式写出模式时，每个线程持有私有的输出适配器实例（各自写入独立的分片文件）以避免锁竞争。当需要写入单一文件时，使用互斥锁或消息通道将数据发送给一个专门的写入线程。
4. **确定性保证**：分片级别的种子独立且可重现，因此无论并行执行多少次，只要总分片数和基础种子固定，生成的数据都将二进制一致。

## 5. 扩展性设计

添加对新数据类型（生成器）的支持只需三步：
1. 在 `generators` 库中新建一个模块，实现 `Generator` 接口。
2. 在生成器注册表中添加一条从名称字符串到该生成器实例的映射。
3. 若新生成器需要独特的参数，只需利用通用参数中的动态字段，无需修改核心接口或调度器。

后处理管道和输出格式的扩展同样基于接口实现，遵循开放-封闭原则。

## 6. 部署与运行环境

- 编译：通过 Cargo 构建系统编译为单个原生可执行文件。
- 依赖：运行时无外部服务依赖，仅需文件系统读写权限。
- 资源占用：通过流式处理和并行控制，内存占用可控；CPU 利用率可通过线程数调节，建议在专用计算节点上运行以快速生成 TB 级数据。

## 7. 总结

StructGen-rs 通过纯 Rust 实现的模块化架构，在保证极高生成性能的同时，提供了清晰的扩展性。其生成的“非语言、富结构”数据能够有效补充传统文本语料，为语言模型的预训练注入规则推理、状态演化和逻辑推演的底层能力。本规格说明书覆盖了系统的全部核心逻辑，可指导后续的详细实现与测试。
